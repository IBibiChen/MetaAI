## 官方文档讲了什么

Spring AI 的 Tool Calling，也叫 Function Calling，解决的是模型无法直接访问外部系统的问题。模型可以根据工具定义提出“调用哪个工具、传什么参数”的请求，但真实执行工具的是
Spring 应用本身，而不是模型

这个边界非常关键：模型永远不应该直接拿到数据库、对象存储、Redis、第三方接口或内部服务的访问权。工具调用的安全责任、参数校验、鉴权、审计、异常处理都在应用侧

### 工具的两类用途

官方把工具主要分成两类：

- 信息检索类工具：查询数据库、Web 服务、文件系统、实时数据或搜索引擎，补足模型不知道的外部信息，也可以用于 RAG 场景
- 动作执行类工具：发送邮件、创建记录、提交表单、触发工作流、执行某个业务动作

这两类工具的风险等级完全不同。信息检索类工具主要关注权限过滤和数据脱敏，动作执行类工具还必须关注幂等、审计、回滚和二次确认

### 工具调用流程

一次完整工具调用通常包含 6 个步骤：

1. 应用把工具定义放入聊天请求，工具定义包括工具名、描述和输入参数 schema
2. 模型判断需要调用工具时，返回工具名和参数
3. 应用根据工具名找到对应工具
4. 应用执行工具并处理工具结果
5. 应用把工具结果回传给模型，或在 `returnDirect=true` 时直接返回调用方
6. 模型基于工具结果生成最终回答

这里最容易犯的错误，是把“模型能调用工具”理解成“模型能执行工具”。真实情况正好相反：模型只做选择和参数生成，执行权必须留在应用侧

## 官方核心概念地图

Spring AI 的工具调用不是单个注解能力，而是一组互相配合的抽象。必须先把这些概念分清楚，否则后面很容易把“给模型看的工具描述”和“应用真正执行的工具逻辑”混在一起

### 工具定义：Tool Definition (`ToolDefinition`)

`ToolDefinition` 表示工具定义，也就是给模型看的工具说明书。它告诉模型当前有哪些工具可以用、每个工具叫什么、适合做什么、输入参数长什么样

它通常包含 3 类信息：

- `name`：工具名称，在一次请求可见的工具集合中必须唯一
- `description`：工具描述，模型靠它判断什么时候调用工具
- `inputSchema`：输入参数 JSON Schema，模型靠它生成工具参数

`ToolDefinition` 的质量会直接影响模型行为。工具名不稳定，模型很难形成稳定调用；描述只写一句“查询数据”，模型无法判断使用边界；输入
schema 把所有字段都设成必填，模型可能为了满足 schema 编造不存在的参数

MetaAI 中 `ToolDefinition` 应遵守 3 个规则：

- 工具名稳定，不要带环境、租户、用户或临时版本信息
- 描述必须写清楚适用场景、禁止场景和是否有副作用
- 参数 schema 只声明模型应该填写的业务参数，安全上下文不要进入 schema

严厉一点说：`ToolDefinition` 写烂了，模型调用错工具不是模型的问题，是你把工具契约设计烂了

### 工具回调：Tool Callback (`ToolCallback`)

`ToolCallback` 表示工具回调，也就是应用侧可执行的工具入口。它是 Spring AI 里真正的工具抽象，核心契约包含 4 类方法：

- `getToolDefinition()`：返回给模型看的工具定义，包括工具名、描述和输入 schema，供模型理解和选择工具
- `getToolMetadata()`：返回工具执行元数据，用于控制 `returnDirect` 等执行行为，不属于模型生成的业务参数
- `call(String toolInput)`：使用模型生成的 JSON 入参执行工具，适合不需要服务端上下文的简单工具
- `call(String toolInput, ToolContext toolContext)`：带服务端上下文执行工具，适合需要 `tenantId`、`userId`、`kbId`
  、权限范围等安全上下文的业务工具

可以把它理解成：

- `ToolDefinition` 负责“让模型知道怎么叫这个工具”
- `ToolMetadata` 负责“让 Spring AI 知道怎么处理这个工具的执行行为”
- `call(...)` 负责“应用怎么真正执行这个工具”
- `ToolContext` 负责“服务端安全上下文怎么进入工具执行过程”

MetaAI 中业务工具默认应优先走带 `ToolContext` 的执行路径。`tenantId`、`userId`、`kbId`、权限范围不能从 `toolInput`
中信任获取，必须从服务端请求上下文或 `ToolContext` 注入，再由工具内部做边界校验

`getToolMetadata()` 不是可有可无的细节。特别是 `returnDirect=true`
会改变工具调用结果的后续处理方式：结果会直接返回调用方，而不是再交给模型生成最终回答。诊断工具和结构化查询工具可以考虑直接返回，需要模型综合解释的工具则不应该轻易开启

如果只理解 `call(String toolInput)`，工具设计很容易退化成“模型参数全可信”。这在多租户系统里是危险设计，因为模型生成的 JSON
入参不能承担租户隔离、用户身份和权限范围的可信来源

Spring AI 内置的 `MethodToolCallback` 和 `FunctionToolCallback` 都是 `ToolCallback` 的具体实现。只有在 MCP、插件系统、工具网关、多
Agent 编排这类扩展场景里，才需要自己实现 `ToolCallback`

### 工具元数据：Tool Metadata (`ToolMetadata`)

`ToolMetadata` 表示工具元数据，也就是控制工具执行行为的元信息。它不是模型业务参数，而是用于告诉 Spring AI 这个工具应该如何被处理

最常见的是 `returnDirect`：

- `false`：工具结果回传给模型，模型再生成最终回答
- `true`：工具结果直接返回给调用方，不再让模型二次加工

`returnDirect=true` 不只是格式选择，它会改变工具调用循环。诊断结果、结构化查询结果、终止型动作结果可以考虑直接返回；需要模型综合解释的结果不要直接返回

`ToolMetadata` 属于 `ToolCallback` 契约的一部分，不能被当成外围备注。尤其是 `returnDirect` 会影响最终响应路径，一旦配置错误，可能出现两类问题：

- 本应由模型解释的工具结果被原样返回，用户看到难懂的内部结构
- 本应直接返回的结构化结果又被模型改写，导致字段丢失或语义漂移

MetaAI 默认策略：诊断工具和纯结构化只读查询可以评估 `returnDirect=true`，普通对话工具保持 `false`，动作工具必须结合确认、审计和返回格式单独设计

### 工具上下文：Tool Context (`ToolContext`)

`ToolContext` 表示工具上下文，也就是服务端注入给工具的安全上下文。它适合传入 `tenantId`、`userId`、`kbId`、权限范围、请求来源等信息

官方文档强调，`ToolContext` 中的数据不会发送给模型。这一点对 MetaAI 非常关键：安全上下文必须从服务端注入，不能让模型生成

`ToolContext` 不等同于工具入参。工具入参来自模型生成的 `toolInput`，会受模型理解、提示词和 schema 影响；`ToolContext`
来自应用侧调用链，适合承载不能交给模型决定的安全信息

正确边界：

- 模型生成业务查询参数
- 服务端通过 `ToolContext` 注入租户、用户和权限信息
- 工具内部基于 `ToolContext` 再次校验边界

错误边界：

- 让模型填写 `tenantId`
- 让模型填写 `userId`
- 让模型决定权限范围
- 让模型选择内部服务地址或凭据

MetaAI 约束：`tenantId`、`userId`、`deptId`、`kbId`、权限范围、API Token、内部 Authorization 不能从 `toolInput`
中信任获取。即使用户在自然语言里提到这些字段，也只能作为待校验的业务线索，最终可信值必须来自服务端上下文

### 工具调用管理器：Tool Calling Manager (`ToolCallingManager`)

`ToolCallingManager` 表示工具调用管理器，也就是协调工具调用生命周期的组件。它的核心职责包括：

- 从 `ToolCallingChatOptions` 中解析 `ToolDefinition`
- 根据模型返回的工具名找到对应 `ToolCallback`
- 执行工具调用
- 处理工具调用结果
- 把结果交回模型或直接返回调用方

如果使用 Spring AI Boot Starter，默认会自动配置 `DefaultToolCallingManager`。普通业务不要急着自定义它，除非你确实需要统一接管工具选择、执行、异常、审计或动态解析逻辑

它的输入可以理解为“模型要求调用某个工具”的请求，输出可以理解为“工具执行后的结果消息或直接响应”。它不应该承载具体业务规则，业务规则应该留在工具方法、领域服务和权限组件里

MetaAI 只有在这些场景才考虑接管或扩展工具调用管理：

- 需要统一记录工具调用审计
- 需要接入审批或二次确认
- 需要动态工具解析和灰度发布
- 需要统一处理工具异常和错误码
- 需要屏蔽或改写部分工具调用结果

### 工具执行：Tool Execution

`Tool Execution` 表示工具执行，也就是模型提出工具调用之后，应用真正执行工具并处理结果的阶段

执行链路可以这样理解：

1. 应用把 `ToolDefinition` 放进请求
2. 模型返回工具名和参数
3. `ToolCallingManager` 找到对应 `ToolCallback`
4. `ToolCallback` 执行业务逻辑
5. 工具结果通过 `ToolCallResultConverter` 转成字符串或结构化结果
6. 结果回传给模型，或在 `returnDirect=true` 时直接返回调用方

工具执行阶段是安全风险最高的地方。所有鉴权、参数校验、租户隔离、幂等、审计和异常处理都必须在这里兜住

### 工具结果转换器：Tool Call Result Converter (`ToolCallResultConverter`)

`ToolCallResultConverter` 表示工具结果转换器，也就是把工具返回值转换成模型可消费内容的转换器。默认情况下，Spring AI 会用
Jackson 把结果序列化成 JSON

不要把它当成无关细节。工具结果会进入模型上下文，返回值越乱，模型越容易误解；返回值越大，越容易浪费上下文窗口；返回值包含敏感字段，就会把内部数据暴露给模型

MetaAI 中工具返回值应该优先使用专用 DTO，并在转换前完成脱敏

默认 Jackson 序列化只能保证“能转成 JSON”，不能保证“适合给模型看”。工程上要主动控制：

- 字段数量，避免把实体对象整包丢给模型
- 字段名称，避免内部数据库字段名污染语义
- 字段长度，避免长文本挤爆上下文窗口
- 敏感字段，避免密钥、内部路径、错误堆栈进入模型上下文
- 空值和异常值，避免模型把内部缺省状态误解成业务事实

MetaAI 的工具结果 DTO 应该是模型消费 DTO，不是接口响应 DTO，更不是持久化实体

### 工具回调解析器：Tool Callback Resolver (`ToolCallbackResolver`)

`ToolCallbackResolver` 表示工具回调解析器，也就是按工具名动态查找工具执行入口的解析器。它用于运行时根据工具名动态解析
`ToolCallback`

适合场景：

- 工具来自 Spring Bean
- 工具由插件系统注册
- 工具数量较多，需要按名称动态加载
- MCP 或外部工具网关需要接入

不适合场景：

- 只有几个固定业务工具
- 工具边界还没稳定
- 权限模型还没设计清楚

动态解析能力很强，但也会放大治理难度。MetaAI 第一阶段不应该为了“显得高级”直接上动态工具注册

## 工具定义方式对比

Spring AI 1.1 提供了两大类工具来源：方法和函数。方法工具偏业务对象，函数工具偏轻量函数式能力

### 工具名称和描述

工具名必须在同一次请求可见的工具集合中唯一。没有显式指定名称时，默认使用方法名

工具描述不是给人看的注释，而是给模型做决策的协议。描述写得含糊，模型就可能不用、误用或在不该调用时调用

一个合格的工具描述至少要说明：

- 什么时候应该使用
- 什么时候禁止使用
- 参数格式和约束
- 返回值代表什么
- 是否会产生副作用

### Methods as Tools

`Methods as Tools` 指把 Java 方法暴露成工具。Spring AI 支持两种方式：

- 声明式：使用 `@Tool`
- 编程式：使用 `MethodToolCallback`

方法工具适合 MetaAI 的大多数项目内能力：

- 查询文档索引状态
- 查询知识库统计摘要
- 查询运行时诊断信息
- 触发受控的业务动作
- 访问已有 Spring Service 或 Repository

方法工具的优势是贴近业务服务边界，能自然使用 Spring Bean、权限组件、配置属性和领域服务。它也更适合处理多个参数、枚举、对象入参和
`void` 返回值

方法工具的限制也要记住：官方文档列出的不支持类型包括 `Optional`、异步类型、响应式类型和函数式类型。需要函数式能力时，应该使用
`Functions as Tools`

`ToolContext` 类型的参数不会成为模型可见输入 schema 的一部分，它是应用侧执行上下文。这个规则对 MetaAI
很重要：工具方法可以同时包含模型入参和服务端上下文，但两者可信级别完全不同

方法工具最佳实践：

- 项目内稳定业务能力优先用方法工具
- 多参数业务工具优先显式声明参数含义
- 涉及权限边界的方法必须带 `ToolContext`
- 不把 Controller 方法直接暴露为工具，工具应该调用领域服务或应用服务

### Declarative Specification: `@Tool`

`@Tool` 是把方法声明成工具的最直接方式，也是 MetaAI 内部稳定工具的默认选择

`@Tool` 主要表达：

- `name`：工具名，不写时默认使用方法名
- `description`：工具描述，强烈建议显式写清楚
- `returnDirect`：工具结果是否直接返回调用方
- `resultConverter`：工具结果转换器

`@ToolParam` 用于补充参数信息：

- `description`：参数含义、格式、取值约束
- `required`：参数是否必填，默认必填

这里有一个很容易被忽略的风险：必填参数如果实际不存在，模型可能为了满足 schema 而编造参数。可选参数必须显式标注为可选，不要让模型被
schema 逼着幻觉

`@Tool` 的描述必须写成协议，而不是写成普通 JavaDoc 摘要。最低要求是包含：

- 适用场景：什么时候应该调用
- 禁止场景：什么时候不应该调用
- 参数语义：参数从哪里来，格式是什么
- 返回值语义：返回内容代表什么
- 副作用：是否会修改状态、触发任务或调用外部系统

`@ToolParam(required=false)` 要谨慎但不能省略。真实可选的参数必须标成可选，否则模型会为了满足必填 schema 编造时间范围、文档
ID 或业务 ID

推荐用于：

- 项目内稳定业务工具
- 只读查询工具
- 诊断工具
- 依赖 Spring Bean 的工具
- 需要清晰 JavaDoc 和注解表达的工具

不推荐用于：

- 工具由外部系统动态提供
- 工具定义需要运行期拼装
- 工具数量很大且需要统一注册中心

### Programmatic Specification: `MethodToolCallback`

`MethodToolCallback` 是把 Java 方法编程式包装成 `ToolCallback`

它适合这些场景：

- 不能或不想在方法上加 `@Tool`
- 工具定义需要在运行期决定
- 工具名、描述、输入 schema 或 metadata 需要统一生成
- 工具来自某个外部注册表或配置中心
- 需要把同一套工具定义批量注册到 `ChatClient` 或 `ChatModel`

不要把 `MethodToolCallback` 当成“更高级所以更好”。如果工具是项目内固定业务方法，用 `@Tool` 更直观、更可维护

编程式定义的价值在于“工具契约由代码集中控制”。例如你需要按租户、环境或灰度策略生成工具集合，或者不能修改原始方法源码时，可以用
`MethodToolCallback` 包装既有方法

MetaAI 里只有这些场景建议使用 `MethodToolCallback`：

- 统一注册一批诊断工具
- 从配置或插件系统生成工具定义
- 对第三方 Java 方法做非侵入式包装
- 需要统一设置 `ToolMetadata` 或 `ToolCallResultConverter`

### Functions as Tools

`Functions as Tools` 指把函数式接口暴露为工具，例如 `Function`、`Supplier`、`Consumer`、`BiFunction`

函数工具适合轻量、无状态、输入输出清晰的能力：

- 格式转换
- 简单计算
- 调用一个单一 adapter
- 查询一个轻量外部服务
- 从 Spring Bean 动态解析函数式工具

函数工具的边界要比方法工具更窄。官方文档列出的函数工具不支持输入或输出 primitive、`Optional`
、集合类型、异步类型和响应式类型。需要复杂参数或集合参数时，优先回到方法工具

函数工具最适合“一个输入对象，一个输出对象”的轻量 adapter。它不适合承载复杂权限校验、跨多个领域服务的业务编排，也不适合直接用
`Map` 这类弱类型结构糊弄 schema

MetaAI 中函数工具可以用于：

- 简单格式化或转换
- 包装单一外部查询 adapter
- 无状态诊断信息聚合

不建议用于：

- 多租户业务查询
- 有副作用动作
- 需要复杂审计的工具
- 需要多个服务协作的业务流程

### Programmatic Specification: `FunctionToolCallback`

`FunctionToolCallback` 是把函数式接口编程式包装成 `ToolCallback`

它适合：

- 把已有 `Function<Request, Response>` adapter 暴露给模型
- 工具逻辑本身无状态
- 工具定义需要编程式生成
- 通过 Spring Bean 动态解析函数工具
- 插件化场景中把函数注册成工具

使用 `FunctionToolCallback` 时，要特别注意输入类型的 schema 质量。函数工具通常只有一个输入对象，输入对象字段必须表达清楚，否则模型会把参数塞错位置

`FunctionToolCallback` 的优雅用法是把函数式 adapter 明确包装成工具，而不是把复杂业务硬塞进
lambda。输入对象要有稳定字段、清晰命名和明确校验规则；输出对象要脱敏、短小、结构稳定

### Dynamic Specification: `@Bean`

Spring AI 还支持把 Spring 容器里的函数式 Bean 按名称动态解析为工具。调用侧可以只传工具名，运行时由 `ToolCallbackResolver`
找到对应 `ToolCallback`

这个能力适合工具注册中心或插件化工具，不适合第一阶段随手使用。默认工具如果配置得太宽，会让模型在不该拥有某个能力的场景里也能看到它

### 定义方式选择原则

| 场景                  | 推荐方式                                     | 原因                      |
|---------------------|------------------------------------------|-------------------------|
| 项目内固定业务工具           | `@Tool`                                  | 简单、直观、可读性最好             |
| 需要包装既有 Java 方法但不改源码 | `MethodToolCallback`                     | 不侵入原方法                  |
| 工具定义运行期生成           | `MethodToolCallback` 或自定义 `ToolCallback` | 可控制名称、schema 和 metadata |
| 无状态函数式 adapter      | `FunctionToolCallback`                   | 输入输出清晰，封装成本低            |
| Spring Bean 函数动态解析  | `@Bean` + `ToolCallbackResolver`         | 适合统一注册和动态选择             |
| MCP、插件、工具网关         | 自定义 `ToolCallback` 或 resolver            | 需要跨系统抽象                 |

MetaAI 第一阶段的优雅方案很明确：内部稳定工具用 `@Tool`，少量动态能力用 `MethodToolCallback`，不要过早引入动态工具注册和
MCP

### 工具注册与可见范围

工具注册方式决定模型在一次请求中能看到哪些工具。工具“可见”本身就是安全边界，不是单纯的代码组织问题

常见注册方式：

- 按请求注册工具：在某次 ChatClient 调用中显式传入工具，适合按场景暴露最小工具集
- 默认注册工具：在 ChatClient 或 ChatModel 默认配置里注册工具，适合低风险、通用、只读能力
- 注册 `ToolCallback`：直接传入已经构造好的工具回调，适合编程式工具定义
- 按工具名动态解析：通过 `ToolCallbackResolver` 在运行时解析工具，适合工具注册中心、插件系统和 MCP

不要把所有工具默认暴露给所有会话。模型看到的工具越多，误调用概率越高；动作工具可见范围越宽，越容易绕过业务入口和权限边界

MetaAI 工具可见范围策略：

- 默认工具集只放只读诊断工具
- 业务查询工具按具体 Controller、ChatClient 或场景显式注入
- 动作工具默认不可见，必须在具备确认、审计和权限校验的专用链路里注入
- 第三方同步、重建索引、删除、覆盖配置等高风险工具不进入普通对话工具集
- 多租户场景中，工具是否可见也要受用户身份、租户权限和业务上下文约束

这个点必须讲狠一点：工具注册不是“把 Bean 放进 Spring 容器”这么简单，而是“给模型发放能力”。能力发放过宽，就是系统边界失守

### 工具执行方式

Spring AI 支持 3 种工具执行模式：

- 框架默认执行：`ChatModel` 通过 `ToolCallingManager` 自动处理工具调用闭环
- `ToolCallAdvisor` 执行：把工具调用纳入 Advisor 链，由 Advisor 管理工具调用循环
- 用户手动执行：关闭内部工具执行后，应用自己检查 `ChatResponse` 中的工具调用并执行

MetaAI 更适合优先使用 `ToolCallAdvisor`。原因很直接：项目里已经存在 Memory、RAG、Logger 等 Advisor 思路，工具调用如果散落在
Controller 或 Service 里，后续很难统一观测、排序和调试

执行模式选择表：

| 执行模式                 | 适用场景                                | MetaAI 建议           |
|----------------------|-------------------------------------|---------------------|
| Framework-Controlled | demo、低风险只读工具、快速验证                   | 可以用于早期验证，但不要放高风险动作  |
| Advisor-Controlled   | 正式对话链路，需要和 Memory / RAG / Logger 协作 | 推荐作为正式工具能力的默认方向     |
| User-Controlled      | 删除、重建、同步、提交、外部动作、需要审批确认             | 高风险动作必须走这个方向或等价确认链路 |

### Framework-Controlled Tool Execution

框架默认执行适合快速验证和简单工具。开发者把工具交给 `ChatClient` 或 `ChatModel`，模型请求工具时，`ChatModel` 内部通过
`ToolCallingManager` 自动执行工具并把结果回传给模型

优点：

- 使用简单
- 代码少
- 适合 demo 和低风险只读工具

问题：

- 工具调用循环隐藏在模型调用内部
- 中间消息不容易暴露给业务侧
- 审计、审批、确认链路不够灵活
- 和复杂 Advisor 链路集成时可观测性较弱

### Advisor-Controlled Tool Execution with ToolCallAdvisor

`ToolCallAdvisor` 会把工具调用实现为 Advisor 链的一部分，并关闭模型内部工具执行。模型请求工具时，由 Advisor
执行工具并继续调用模型，直到不再需要工具

官方文档强调它的优势包括：

- 可观测性更好，其他 Advisor 可以观察工具调用迭代
- 能和 Chat Memory Advisor 更好协作
- 可扩展性更强，方便定制工具调用行为

MetaAI 如果要把工具能力做成正式能力，优先考虑这个模式。因为当前项目已经有 Memory、RAG、Logger、Retrieval Advisor
相关设计，工具调用必须纳入同一条编排链路，否则后续排查会非常痛苦

### User-Controlled Tool Execution

用户手动执行是最高控制权模式。通过设置 `internalToolExecutionEnabled=false`，应用自己检查模型响应里的工具调用，再调用
`ToolCallingManager` 或自定义逻辑执行工具

适合：

- 高风险动作工具
- 需要人工确认
- 需要审批流
- 需要多阶段执行
- 需要强审计和回滚
- 需要向前端展示“模型准备调用某工具，是否确认”

不适合：

- 简单只读工具
- 低风险诊断工具
- 第一阶段快速验证

MetaAI 中删除、重建索引、批量同步第三方文档、修改配置这类能力，如果未来要工具化，必须走用户手动执行或至少走显式确认链路。把这些能力直接交给框架默认执行，是非常危险的设计

`internalToolExecutionEnabled=false` 的价值不是“多写代码”，而是把工具执行权交还给应用。应用可以在真正执行前做确认弹窗、权限复核、审计记录、参数脱敏、幂等检查和风险提示

### 工具上下文示例：ToolContext

`ToolContext` 用于把服务端上下文传给工具，例如 `tenantId`、`userId`、`kbId`、权限范围等

官方文档明确说明，`ToolContext` 里的数据不会发送给模型。这正适合 MetaAI 的多租户和知识库隔离场景：模型可以生成业务参数，但不能生成安全上下文

正确做法：

```java

@Tool(description = "查询指定客户信息，必须在当前租户范围内执行")
Customer getCustomerInfo(Long id, ToolContext toolContext) {
    return customerRepository.findById(id, toolContext.getContext().get("tenantId"));
}
```

错误做法：

```java

@Tool(description = "查询指定租户下的客户信息")
Customer getCustomerInfo(String tenantId, Long id) {
    return customerRepository.findByTenantIdAndId(tenantId, id);
}
```

第二种写法把 `tenantId` 交给模型填写，这在多租户系统里是严重设计错误

### 直接返回：returnDirect

默认情况下，工具结果会回传给模型，再由模型生成最终回答。`returnDirect=true` 表示工具结果直接返回给调用方，不再让模型二次加工

适合 `returnDirect=true` 的场景：

- 结果已经是结构化业务响应
- 工具结果不需要模型润色
- 工具是一个终止型动作
- RAG 工具只想直接返回召回结果或诊断信息

不适合的场景：

- 需要模型综合多个工具结果
- 需要自然语言解释
- 工具结果包含敏感字段，需要应用先脱敏

### 异常、观测与审计

工具执行失败时，Spring AI 会把异常包装到工具执行异常链路中。可以通过 `ToolExecutionExceptionProcessor` 统一决定：

- 把错误信息转成文本返回给模型
- 直接抛出异常交给调用方处理

`spring.ai.tools.throw-exception-on-error` 可以控制默认处理策略。动作类工具建议倾向抛出异常，由应用统一处理，避免模型把失败结果“圆回来”

Spring AI 的工具调用还支持观测能力，会记录耗时和 tracing 信息。工具参数和结果默认不作为 span 属性导出，这是合理的，因为工具参数很可能包含用户数据、业务
ID 或内部上下文

异常处理不要把内部异常原文直接交给模型。模型不需要看到数据库连接串、Redis 命令、对象存储路径、堆栈信息或内部服务
Authorization。工具异常应该转成稳定错误码和脱敏说明

MetaAI 工具日志建议记录：

- 工具名
- 调用来源和会话 ID
- 用户、租户和知识库标识的脱敏摘要
- 参数摘要，不记录完整敏感内容
- 执行耗时
- 执行状态
- 错误码和脱敏错误原因
- 关联业务对象 ID

不建议记录：

- API Key、Token、Authorization
- 数据库密码、Redis 密码、对象存储密钥
- 完整文件正文或知识库 chunk 内容
- 未脱敏的工具返回结果
- Java 堆栈全量文本进入模型上下文

动作工具必须审计。只打普通日志不够，因为日志不等于业务审计。审计至少要能回答：谁让模型调用了哪个工具，参数是什么，结果是什么，对哪个业务对象产生了影响，是否经过确认

## 推荐学习路径

### 第一步：先学 ChatClient

不要一上来就写工具。工具调用依附在 ChatClient 或 ChatModel 调用链上，如果连 prompt、system message、user message、同步响应、流式响应和
Advisor 的基本关系都没搞清楚，直接写工具只会堆出一堆不可控入口

学习目标：

- 会创建基础 `ChatClient`
- 理解一次 `prompt().call().content()` 的执行流程
- 理解默认 Advisor 和运行时 Advisor 的差异
- 理解模型参数、系统提示词和用户输入的边界

### 第二步：学习 `@Tool` 和 `@ToolParam`

先写只读工具，不要先写动作工具

建议练习：

- 查询当前时间
- 查询系统运行摘要
- 查询知识库索引状态
- 查询某个文档的处理状态

学习目标：

- 工具名如何生成
- 工具描述如何影响模型是否调用
- 参数 schema 如何生成
- 参数格式错误时如何处理

### 第三步：重点学习 ToolContext

MetaAI 的关键不是“模型会不会调用工具”，而是“模型调用工具时有没有越权”

学习目标：

- `tenantId` 不由模型生成
- `userId` 不由模型生成
- `kbId` 可以来自明确业务请求，但必须经过权限校验
- 权限范围通过服务端上下文传入工具
- 工具内部再次做边界校验，不信任模型参数

### 第四步：学习 ToolCallAdvisor

当工具调用进入真实业务链路时，应该纳入 Advisor 体系

学习目标：

- 工具调用和 Chat Memory 如何协作
- 工具调用和 Logger Advisor 如何观测
- 工具调用和 RAG Advisor 的先后顺序如何设计
- `conversationHistoryEnabled` 和 `.disableMemory()` 的使用边界

### 第五步：最后学习 MCP

MCP 适合跨进程、跨语言、外部工具生态或插件化工具发现。不要一开始就用 MCP 重构内部 Java 工具，这属于过早复杂化

适合引入 MCP 的信号：

- 工具不在当前 JVM 内
- 工具由其他团队维护
- 工具需要动态发现
- 工具需要跨语言复用
- 工具数量和权限模型已经复杂到需要独立治理

## MetaAI 最佳实践

### 不要默认把 RAG 检索改造成 Tool

MetaAI 已经有 RAG 链路和 `RetrievalAugmentationAdvisor` 思路。RAG 的核心是稳定地把检索上下文注入模型，而不是让模型自由决定要不要检索

如果把 RAG 默认包装成普通工具，会引入几个问题：

- 模型可能在该检索时不检索
- 模型可能在不该检索时乱检索
- 检索参数更容易被模型污染
- 原有 RAG 可观测链路会被工具调用链路稀释

更稳妥的做法：

- 普通知识库问答继续走 RAG Advisor
- 工具调用只负责补充明确的业务能力
- 只有在构建“多能力 Agent”时，才把知识库搜索作为候选工具之一

### 工具按风险分类

建议在代码和文档上把工具分成 3 类：

- `QueryTool`：只读查询工具，例如查询文档状态、索引状态、会话摘要
- `CommandTool`：动作工具，例如触发重新索引、创建同步任务、提交外部系统操作
- `DiagnosticTool`：诊断工具，例如查看当前模型、向量库、Redis index、配置摘要

这不是命名洁癖，而是风险治理。只读工具可以更早开放，动作工具必须晚一点，并且要加审计和确认机制

### 默认只开放只读工具

第一阶段只允许只读工具进入模型可见工具集

动作工具必须满足这些条件后再开放：

- 已完成鉴权
- 已完成参数校验
- 已完成幂等设计
- 已有操作审计
- 已明确失败回滚或补偿策略
- 对高风险动作有二次确认机制

不要让模型直接删除知识库、清空索引、覆盖配置或同步第三方文件。让模型直接碰这些动作，是把系统稳定性赌在一次自然语言推理上，风险很高

### 工具暴露策略

MetaAI 应按阶段暴露工具，而不是一次性把所有能力交给模型

第一阶段：只读诊断工具

- 查询当前模型 provider
- 查询向量库类型
- 查询 RAG 默认参数
- 查询关键开关状态
- 不访问用户私有正文
- 不修改任何状态

第二阶段：只读业务查询工具

- 查询文档索引状态
- 查询最近失败原因
- 查询知识库文件数量摘要
- 查询外部同步任务状态
- 必须带 `ToolContext` 做租户、用户和权限过滤

第三阶段：带确认和审计的动作工具

- 触发单文档重新索引
- 触发外部文档同步
- 创建异步排查任务
- 必须有鉴权、幂等、审计、确认和失败补偿策略

禁止直接暴露到普通对话的工具：

- 删除知识库
- 清空向量索引
- 批量重建索引
- 修改模型、向量库或存储配置
- 读取密钥、Token、连接串或内部 Authorization
- 绕过业务权限读取用户私有内容

### 安全上下文只能来自服务端

以下字段不允许由模型生成：

- `tenantId`
- `userId`
- `deptId`
- 权限范围
- API Token
- 数据源凭据
- 内部服务地址

这些字段必须来自登录态、请求上下文、服务端配置或 `ToolContext`

模型可以填写的问题参数：

- 查询关键词
- 文档名称
- 时间范围
- 排序偏好
- 用户自然语言意图中明确出现的业务 ID

即使模型填写了业务 ID，工具内部也必须基于服务端上下文做权限校验

### 工具描述按协议写

不推荐：

```
@Tool(description = "查询文档")
```

推荐：

```java

@Tool(description = """
        查询当前用户有权限访问的知识库文档处理状态
        仅用于用户询问文档是否上传成功、是否完成索引、失败原因是什么的场景
        禁止用于回答文档正文内容，正文问答应走 RAG 检索链路
        documentId 来自用户明确提供的文档 ID，租户和用户权限由服务端上下文注入
        返回文档索引状态、失败原因和最近更新时间，不修改任何业务状态
        """)
DocumentIndexStatus queryDocumentIndexStatus(String documentId, ToolContext toolContext) {
    // ...
}
```

描述要像工具协议，而不是一句自然语言备注

### Advisor 顺序要显式设计

建议优先让工具调用进入统一 ChatClient 配置，而不是每个 Controller 临时 `.tools(...)`

一个可行方向：

- Memory Advisor 负责会话历史
- RAG Advisor 负责知识库上下文增强
- ToolCallAdvisor 负责工具调用循环
- Logger Advisor 负责请求、响应和排查日志

实际顺序要结合当前 ChatClient 配置验证，不能凭感觉乱放。尤其是 ToolCallAdvisor 和 Memory Advisor 同时存在时，要明确是否让
ToolCallAdvisor 自己维护内部历史，避免重复记忆或上下文膨胀

### 工具返回值要稳定

工具返回值不要直接返回复杂实体对象，尤其不要把 JPA / MyBatis 实体原样暴露给模型

推荐返回专门的 DTO：

- 字段少
- 语义稳定
- 不包含凭据和内部路径
- 不包含数据库实现细节
- 不包含无关大字段

工具结果是模型的输入材料，不是后端接口响应的随手复用

### 动作工具必须可审计

任何会修改状态的工具都必须记录：

- 谁发起
- 从哪个会话发起
- 模型原始意图是什么
- 工具名是什么
- 参数是什么
- 执行结果是什么
- 失败原因是什么
- 关联的业务对象是什么

如果这些信息不记录，后续出现误操作时无法追责，也无法改进工具描述和权限策略

## 最小落地练习

第一步不要做“删除索引”“重新同步”这类动作工具。建议先做 `MetaDiagnosticTools`

目标：让模型能够回答当前 MetaAI 运行环境的诊断类问题，但不修改任何业务状态

建议工具：

```java

@Tool(description = """
        查询当前 MetaAI 运行环境的诊断摘要
        仅用于排查模型 provider、向量库类型、检索默认参数、关键开关状态
        禁止用于查询用户私有数据、知识库正文内容或执行任何修改动作
        租户、用户和请求来源由服务端上下文注入
        返回脱敏后的运行摘要，不包含 API Key、密码、Token 或内部凭据
        """)
MetaDiagnosticSummary inspectRuntime(ToolContext toolContext) {
    // 从配置属性和已装配 Bean 中读取脱敏摘要
}
```

`MetaDiagnosticSummary` 建议只包含：

- 当前 chat provider
- 当前 embedding provider
- 当前 vector store 类型
- RAG 默认 `topK`
- RAG 默认 `similarityThreshold`
- query transformer 是否启用
- post processor 是否启用
- snapshot 是否启用
- 外部适配器是否启用

不要包含：

- API Key
- Redis 密码
- 数据库密码
- 对象存储密钥
- 完整 JDBC URL
- 内部服务 Authorization

第二步可以做只读业务工具：

- 查询文档索引状态
- 查询最近一次索引失败原因
- 查询知识库文档数量摘要

第三步再考虑动作工具：

- 触发单文档重新索引
- 触发外部文档同步
- 创建异步排查任务

动作工具必须先补齐鉴权、幂等和审计，不满足这些条件就不要开放给模型

## 推荐落地顺序

1. 建立 `tools` 包，先放只读诊断工具
2. 为工具返回值定义专用 DTO，不复用持久化实体
3. 在 ChatClient 配置中统一注册工具或工具解析器
4. 引入 `ToolContext`，把 `tenantId`、`userId`、`kbId`、权限范围从服务端传入
5. 增加工具调用日志，至少记录工具名、脱敏参数、耗时和执行结果
6. 对动作工具增加审计表或审计日志
7. 再评估是否引入 `ToolCallAdvisor`
8. 最后评估 MCP，不要提前把内部工具拆成外部协议

## 测试建议

工具调用测试不能只测 Java 方法本身，还要测模型可见工具定义后的行为边界

至少覆盖：

- 工具方法单元测试
- 参数为空或格式错误
- 模型传入越权业务 ID 时被拒绝
- `ToolContext` 缺失必要上下文时被拒绝
- 只读工具不产生状态变更
- 动作工具重复调用时保持幂等
- 工具异常时不会泄露内部堆栈、密钥或连接信息
- `returnDirect=true` 的工具不会再被模型二次加工

如果使用 `ToolCallAdvisor`，还要验证：

- 与 Memory Advisor 同时存在时不会重复写入历史
- 与 RAG Advisor 同时存在时上下文顺序符合预期
- Logger 不输出敏感工具参数和结果

## 判断是否该做成工具

可以用下面的问题做决策：

- 这个能力是否需要模型根据语义自主选择
- 这个能力是否需要访问模型不知道的实时信息
- 这个能力是否有明确输入、明确输出和稳定边界
- 这个能力失败时是否能安全恢复
- 这个能力是否已经有鉴权、校验和审计

如果答案是否定的，就先不要做成工具。很多能力更适合做成普通后端接口、RAG Advisor、定时任务或人工触发的运维动作

## 核心结论

Spring AI Tool Calling 不是“给模型加几个 Java 方法”这么简单。它本质上是把模型的自然语言推理能力接入真实系统能力，因此必须按系统边界来设计

最基础的判断必须先站稳：`ToolDefinition` 是模型契约，`ToolCallback` 是应用执行入口，`ToolContext` 是安全上下文边界，
`Tool Execution` 是风险控制现场

MetaAI 当前最优雅的方案是：

- RAG 问答继续走现有 RAG Advisor 链路
- 工具调用从只读诊断工具开始
- 安全上下文通过 `ToolContext` 注入
- 动作工具必须先完成鉴权、幂等和审计
- 成熟后再用 `ToolCallAdvisor` 统一纳入 ChatClient Advisor 链
- MCP 放到跨系统、插件化或动态工具发现阶段再考虑

安全默认值必须固定下来：

- 不信任 `toolInput` 中的租户、用户和权限字段
- 不默认暴露所有工具
- 不把实体对象直接返回给模型
- 不把工具参数和结果全量写日志
- 不把内部异常原文交给模型
- 不把 RAG 检索粗暴改造成普通 Tool

别把工具调用当作功能捷径。它会放大系统能力，也会放大系统漏洞。越早把边界、权限和审计设计清楚，后面越不容易踩坑
